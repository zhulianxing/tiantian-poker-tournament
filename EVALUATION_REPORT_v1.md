## 三、重新开发 vs 二次开发对比

### 方案 A：重新开发（从零开始）

**优点**
- 可统一技术栈（如全 TypeScript / Flutter 跨端）
- 架构可原生支持多语言/多区域
- 代码风格统一，无历史包袱

**缺点**
- **工期**：至少 3-4 周重写已验证的核心逻辑
- **风险**：SNG 引擎边界条件（All-In/短筹码/无限循环）需重新踩坑
- **浪费**：已完成的 E2E 测试、Bug 修复、真机验证全部作废
- **成本**：相当于把 75% 的进度清零

**结论**：❌ 不推荐

---

### 方案 B：二次开发（在当前基础上扩展）

**优点**
- **工期**：1-2 周即可上线产品化版本
- **风险低**：核心引擎已验证，只需包装和扩展
- **保留资产**：数据库、API、APP、测试报告全部保留
- **渐进式**：可边运营边迭代，快速验证市场

**缺点**
- 部分代码风格不统一（JS + Kotlin 混合）
- 需要补 i18n 架构
- 单机架构需扩展为多区域

**结论**：✅ **强烈推荐**

---

## 四、核心建议：二次开发

### 理由

1. **扑克引擎是核心壁垒** — 重新写 SNG 状态机、牌型判定、边池计算、Bot AI 需要 1-2 周且容易引入新 Bug
2. **已验证的 E2E 流程** — 从注册→入座→开赛→打牌→结算的完整链路已通过测试，重写意味着重新测试
3. **Android APP 已编译通过** — Jetpack Compose 是现代化技术栈，无需更换
4. **数据库设计合理** — 9 张表覆盖业务需求，只需加字段（如多语言）无需重构

### 需要补强的部分（不是重写）

| 补强项 | 工作量 | 说明 |
|--------|--------|------|
| 多语言 i18n | 2-3 天 | Android strings.xml + 后端多语言字段 + 官网 |
| 多服务器架构 | 2-3 天 | 首尔主服 + 新加坡/东京边缘节点 + 美国下载/备份 |
| APP UI 美化 | 3-4 天 | 不是重写，是皮肤/动效升级 |
| 官网建设 | 3-4 天 | Next.js + Tailwind，全新但独立 |
| Admin 后台 | 2-3 天 | 补全缺失模块 |
| 支付生产配置 | 1 天 | 填参数 + 测试 |

**总补强工期：约 2 周**，vs 重新开发 **4 周+**

## 五、多语言版本方案

### 目标语言

| 语言 | 优先级 | 对应服务器 |
|------|--------|-----------|
| 中文（简体） | P0 | 首尔（主） |
| 英文 | P0 | 美国 |
| 韩文 | P1 | 首尔/东京 |
| 日文 | P1 | 东京 |
| 繁体中文 | P2 | 新加坡 |

### 技术方案（二次开发兼容）

**Android APP**
- 使用 Android 原生 `values-zh-rCN` / `values-en` / `values-ko` / `values-ja` 资源目录
- 当前 `strings.xml` 已有中文基础，只需翻译
- 语言切换跟随系统，或 APP 内手动切换

**后端**
- `venues` 表加 `locale` 字段
- `players` 表加 `preferred_language` 字段
- 错误码/提示语用 i18n key，前端按语言渲染

**官网**
- Next.js i18n 路由（`/zh` `/en` `/ko` `/ja`）
- 按 IP 自动跳转最近节点

**数据库**
- 牌桌名称、赛事说明、规则文案等用户可见内容加 `locale` 字段

**架构原则**：
- **首尔是唯一数据权威**：所有写操作、支付、分账、牌局逻辑都在首尔
- **边缘节点只读/转发**：新加坡/东京/美国只做 Nginx 反向代理 + WebSocket 转发 + 静态资源
- **降低延迟**：亚洲玩家连新加坡/东京，欧美玩家连美国，再通过专线回首尔
- **DB 备份**：首尔 → 美国每日备份
